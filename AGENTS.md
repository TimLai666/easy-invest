## 專案定位

Easy Invest 是以台股為主、API-first 的投資建議系統。它的目標是根據使用者庫存、交易流水、配息與市場資料，產生可追蹤、可解釋的投資建議，例如「應該賣出哪些標的、買進哪些標的、調整到什麼目標比例」。

預設情境下，本系統只提供建議，不直接替使用者買賣。未來若串接券商 API，必須先支援唯讀查庫存與對帳，再另外設計明確授權、風控、審計與停用機制，才可以討論自動下單。

## 技術方向

- 後端使用 Go。
- 系統必須提供 API，且以 API 為主；UI 只是 API 的操作介面。
- UI 是類儀表板，可新增交易、檢視庫存、看建議、做設定與對帳操作。
- 資料庫使用 PostgreSQL。
- 本地與部署流程以 Docker Compose 為基礎。
- 分析與資料處理盡量使用 Insyra。
- 台股優先，架構保留未來接虛擬貨幣與其他資產類別的空間。

## 資料與投資邏輯原則

- 庫存不能只做 CRUD；庫存必須由交易流水、公司行動與調整紀錄計算出來。
- 同一支股票允許多筆不同買入時間、不同價格、不同股數或張數的批次。
- 一張台股等於 1000 股。API 與 UI 都必須清楚標示輸入單位，避免股與張混用。
- 歷史交易不可為了對齊券商資料而直接改掉。若成本、股數或現金因配息、配股、減資、分割、手續費、交易稅或券商算法差異而需要修正，必須新增可追蹤的調整事件。
- 建議輸出必須能回溯資料來源、策略版本、輸入狀態、計算時間、假設條件與風險限制。
- 策略與回測應共用同一套核心邏輯；差別只在輸入資料來源與輸出落點。
- 策略核心不得直接做網路請求、資料庫讀寫或檔案 I/O。資料先由外層整理成輸入，再交給策略計算。

## API Key 與安全

- 使用者可在 UI 建立本系統的 API key，並用 key 操作自己的資料，例如新增交易、查庫存、取得建議。
- API key 只能儲存雜湊值，不得明文保存；必須支援撤銷、輪替、權限範圍與最後使用時間。
- 未來券商 API 憑證與本系統 API key 要分開處理，不可混用。
- 券商 API 串接第一階段以唯讀查庫存與成交紀錄為主。若要支援下單，必須另行設計權限、雙重確認、最大委託金額、停損停用、審計紀錄與緊急停止。

## 資料來源要求

- 台股資料優先使用官方或授權來源，例如 TWSE、TPEx、公開資訊觀測站，以及明確授權的資料商。
- 每筆市場資料應記錄來源、抓取時間、資料時間、版本或修正狀態。
- 不可靠或未授權的爬蟲資料不得成為唯一真源。
- 建議結果要標示使用的市場資料時間，避免把延遲資料誤認為即時狀態。

## Insyra 上游規則

- 實作分析、指標、時間序列、回測或資料轉換前，先確認 Insyra 是否已有合適能力。
- 若 Insyra 做不到或不好用，先到 Insyra 開 upstream issue。
- 對應程式碼必須留下 `TODO(insyra#<issue-number>): ...`，說明目前暫時方案與等待上游改善的內容。
- 同步更新 `docs/insyra-upstream-todo.md`，列出 issue 連結、受影響功能、目前替代做法與移除 TODO 的條件。
- 不要在本專案默默長出一套大型替代分析框架，除非使用者明確同意。

## 我建議先修正原始文件的地方

- 原本參考文件偏向「SaaS 直接下單 + LocalAgent 執行」。本專案應先做建議系統，不預設自動下單。
- 原本參考文件主張正式資料庫只用 AutoMigrate。這對正式投資資料風險偏高；進入實作時，建議採用可審查的 migration 流程，至少生產環境不可只靠 AutoMigrate。
- 原本參考文件未完整處理配息、配股、減資、股數/張數單位與券商成本對帳。本專案必須把交易流水與對帳模型放在核心。
- 原本參考文件以 A 股與黃金 ETF 為例。本專案以台股為主，所有交易日曆、股價單位、手續費、交易稅、配息與公開資訊資料源都要改成台灣市場語境。

## 待完善清單

- 明確決定正式環境的資料庫 migration 工具與流程。
- 明確定義券商對帳規則：券商成本、已實現損益、現金股利、股票股利、減資與零股如何對齊。
- 明確定義即時、近即時、日終資料各自可用於哪些建議情境。
- 建立 Insyra 缺口 issue 清單與 TODO 編號。
- 建立台股交易日曆、除權息與公司行動資料來源策略。
- 建立未來虛擬貨幣 connector 的資料模型，避免把股票專屬欄位寫死在核心模型。
- 補上投資建議的風險揭露與合規文字，避免讓 UI 呈現成保證獲利或代客操作。
