# 產品需求

## 目標

Easy Invest 要做的是「投資建議系統」，不是一開始就替使用者自動下單的交易機器人。使用者輸入或同步自己的庫存與交易紀錄後，系統依照策略邏輯、市場資料與風險限制，產生可解釋、可追蹤的調整建議。

核心使用情境：

1. 使用者新增自己持有的股票、買入日期、價格、股數或張數。
2. 同一支股票可以有多筆不同買入時間與成本的批次。
3. 使用者日後買進、賣出、收到配息、配股、減資或需要對帳時，都新增一筆事件，不直接改庫存結果。
4. 系統根據最新庫存、成本、可用現金與市場資料，產生投資建議。
5. 使用者若真的買賣，回到系統新增交易，系統再用新的狀態重新計算建議。

## 非目標

- 不保證獲利。
- 不預設自動買賣。
- 不以爬蟲資料當唯一真源。
- 不把使用者券商帳戶的狀態直接覆蓋成本系統。
- 不把 UI 做成主要整合介面；所有功能必須先能透過 API 完成。

## 使用者與權限

使用者可在 UI 產生本系統的 API key。API key 用於操作自己的資料，例如新增交易、查詢庫存、取得建議、查詢建議歷史。

API key 應具備：

- 名稱與用途描述。
- 權限範圍，例如唯讀、交易紀錄寫入、建議讀取、設定管理。
- 建立時間、最後使用時間、停用時間。
- 可撤銷與可輪替。
- 只儲存雜湊值，不保存明文。

## 建議輸出

每次建議都必須留下快照，包含：

- 建議 ID。
- 使用者 ID。
- 策略版本。
- 市場資料時間。
- 庫存快照時間。
- 建議動作：買進、賣出、持有、暫不動作。
- 標的、建議數量、建議金額、目標比例。
- 推薦理由與主要風險。
- 使用的資料來源。
- 系統信心或訊號強度。
- 建議狀態：草稿、已查看、使用者採納、使用者忽略、已過期。

建議不能只輸出「買」或「賣」。必須說明原因、資料時間與限制，避免使用者把舊資料當成即時訊號。

## 台股優先

第一階段以台股為主：

- 上市股票與 ETF。
- 上櫃股票與 ETF。
- 零股與整股都要保留模型空間。
- 一張等於 1000 股。
- 手續費、交易稅、股利、配股、除權息與減資都要在資料模型中留位置。

未來要能擴充到虛擬貨幣，所以核心模型不要把「交易所、交易日、股利、張」等股票專屬概念寫死在所有層級。股票專屬欄位應放在資產類型或 connector 的延伸資料中。

## 資料來源策略

資料來源優先順序：

1. 官方來源：TWSE、TPEx、公開資訊觀測站等。
2. 明確授權的資料商。
3. 券商 API 回傳的使用者帳務與成交資料。
4. 其他來源只能做輔助，不可當唯一真源。

系統要記錄每批資料的來源、抓取時間、資料所屬時間、修正版本與匯入狀態。投資建議必須引用已入庫、可追蹤的資料，不應在策略核心裡即時抓網路資料。

## 未來券商 API

券商串接分階段：

1. 唯讀查庫存。
2. 唯讀查成交紀錄與帳務明細。
3. 協助對帳，產生差異與建議調整事件。
4. 在使用者明確授權與完整風控後，才討論下單能力。

券商 API 憑證與本系統 API key 必須分開管理。未來如果儲存在伺服器端，必須加密、限制權限、完整審計，並提供停用機制。

## 策略方向

參考文件中的核心思路可以保留：

- 用目標權重與目前權重的差距推導調整建議。
- 用回測檢查策略是否優於基準，例如定期定額或使用者目前持有策略。
- 用 GA 或其他最佳化方法尋找策略參數。
- 策略與回測共用同一套決策核心。

但本專案要先把「建議」與「下單」切開。策略核心輸出的應是 recommendation intent，不是券商委託單。

## 待討論決策

- 正式資料庫 migration 工具。
- 台股即時行情、日 K、財報、除權息資料的最終來源組合。
- 成本算法要以券商顯示為準，或以系統內部可解釋算法為準，再用調整事件對齊。
- Insyra 對台股時間序列、技術指標、回測與投資組合績效計算的覆蓋程度。
- 未來自動下單是否需要 LocalAgent，或只由伺服器端加密管理券商憑證。
