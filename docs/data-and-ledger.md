# 交易流水與庫存模型

## 核心想法

庫存不是一張可以任意修改的表。庫存是由一連串事件計算出來的結果。

事件包含：

- 買進。
- 賣出。
- 現金股利。
- 股票股利。
- 除權、除息。
- 股票分割。
- 減資。
- 手續費。
- 交易稅。
- 匯入券商成交紀錄。
- 對帳調整。

歷史事件一旦建立，原則上不直接修改。若需要修正，新增反向或調整事件，並保留原因、時間與操作者。

## 單位

台股同時支援股與張：

- `share`：股。
- `lot`：張。
- 1 張 = 1000 股。

API 需要明確要求使用者傳入單位。內部計算統一轉成股，UI 可以顯示成股或張。

## 買進批次

同一支股票可以有多個批次：

| 批次 | 日期 | 價格 | 數量 |
|---|---|---:|---:|
| A | 2026-01-10 | 100 | 1000 股 |
| B | 2026-02-15 | 90 | 2000 股 |
| C | 2026-03-20 | 110 | 500 股 |

系統需要保留批次資訊，因為不同批次會影響成本、已實現損益、建議賣出順序與稅費估算。

## 賣出規則

賣出時要能指定成本沖銷方法：

- FIFO：先買先賣。
- LIFO：後買先賣。
- Average：平均成本。
- Specific Lot：指定批次。

第一階段可以先實作 FIFO 或平均成本，但資料模型必須保留指定批次的空間，因為券商顯示成本可能與內部算法不同。

## 配息與成本

現金股利可能影響使用者理解的投資成本，但不應直接改掉原始買進價格。

建議保存兩套概念：

- 原始成本：買進價格、手續費、交易稅等原始事件推導。
- 調整後成本：納入現金股利、股票股利、減資或券商對帳調整後的成本視角。

這樣可以同時回答：

- 我當初買這批花了多少錢？
- 扣掉配息後，我的券商顯示成本為什麼變成這樣？
- 這次建議賣出是用哪一種成本觀點？

## 對帳調整

券商資料不是直接覆蓋內部資料，而是形成 broker snapshot。系統比對後產生 reconciliation run。

差異類型：

- 股數不一致。
- 平均成本不一致。
- 批次成本不一致。
- 現金股利漏記。
- 股票股利漏記。
- 手續費或交易稅不同。
- 使用者手動輸入錯誤。

使用者確認差異後，系統新增 adjustment event。調整事件要包含：

- 對應 broker snapshot。
- 調整原因。
- 調整欄位。
- 調整前後數值。
- 是否影響成本、股數或現金。
- 建立者與建立時間。

## Ledger Event 建議欄位

概念欄位：

- `id`
- `user_id`
- `asset_id`
- `event_type`
- `trade_date`
- `settlement_date`
- `quantity_shares`
- `price`
- `gross_amount`
- `fee`
- `tax`
- `currency`
- `source`
- `source_ref`
- `lot_id`
- `notes`
- `created_at`
- `voided_at`
- `void_reason`

`event_type` 至少包含：

- `buy`
- `sell`
- `cash_dividend`
- `stock_dividend`
- `split`
- `capital_reduction`
- `fee_adjustment`
- `tax_adjustment`
- `broker_reconciliation_adjustment`
- `manual_correction`

## 快照

Portfolio snapshot 是計算結果，不是唯一來源。

快照用途：

- 加速查詢。
- 保存建議當下的庫存狀態。
- 支援回放建議。
- 支援對帳比較。

若 ledger event 有變動，相關快照要標示過期或重算。

## 建議引擎需要的輸入

策略核心不直接讀資料庫。外層要先整理成輸入：

- 目前庫存。
- 各批次成本。
- 可用現金。
- 使用者風險設定。
- 市場價格與歷史資料。
- 股利與公司行動資料。
- 策略參數。

策略核心輸出 recommendation intent，由 API 層保存成 recommendation run。
